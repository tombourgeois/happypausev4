APP CONCEPT
HappyPause helps users work in focus cycles and take structured, guided pauses ( “HappyPauses”) that suggest certain activities.

Default timing:
- Focus session (Y) = 55 minutes
- HappyPause (X) = 5 minutes
- Both values editable in Settings

When Focus ends, the app automatically launches a HappyPause with a recommended activity. When the HappyPause is completed, we revert to the focus which starts automatically.  Logic: Focus Timer starts -> Focus Timer ends or “Have a HappyPause” button pressed ->Happypause timer starts -> HappyPause timer Ends or HappyPause completed or Happy Pause Skipped -> Loop back to Focus Timer Starts…

THUMBS UP/DOWN FEEDBACK SYSTEM (IMPORTANT)

Each activity supports preference learning, a bit like an “Up vote” and “Down vote” feature that determines if the activity will be more often or less often in rotation. This is represented by the ThumbsPref value which is set by default at zero, and cannot be less than -10 and more than 10. For each increment in value, the odds double: so if the ThumbsPref = 1, then it means the activity has twice the chance of being selected. If ThumbsPref = 2, then it means the activity has thrice the chance of being selected… and so forth. Same goes for the negative values: so if the ThumbsPref = -1, then it means the activity has twice less chances of being selected. If ThumbsPref = 2, then it means the activity has thrice less chances of being selected…

When user taps:
- Thumbs Up → log happypause_thumb_up  && increase [Activity]ThumbsPref by one
- Thumbs Down → log happypause_thumb_down && decrease [Activity]ThumbsPref by one

These values are shown next to each thumb icon. If the value is positive, the token icon next to the Thumbs up increases, if the value is negative, the the token icon next to the Thumbs down increases.

How the Thumbs mecanic work:
1.	Each activity has two tracking fields:
•	Score (integer), default 0, clamped between -10 and +10.
•	LastShownAt (timestamp or empty if never shown).
The app also keeps a RecentList containing the IDs of the last 4 activities shown (most recent first).
2.	Voting updates the score:
•	When the user taps Thumbs Up, increase the activity’s Score by +1.
•	When the user taps Thumbs Down, decrease the activity’s Score by -1.
•	After any change, clamp the Score so it never goes above +10 or below -10.
3.	Convert the score into a base “weight” (this weight is the activity’s probability relative to others):
•	If the Score is 0, the activity’s base weight is 1 (neutral, equal chance).
•	If the Score is positive, the base weight is Score + 1.
This matches the rule: +1 → 2×, +2 → 3×, +3 → 4×, etc.
•	If the Score is negative, the base weight is 1 / (|Score| + 1).
This matches the rule: -1 → 1/2 (two times less), -2 → 1/3, -3 → 1/4, etc.
4.	Apply the “shown within the last 2 hours” moderation (reduce the effect of thumbs by half):
•	Compute the score’s raw multiplier magnitude as (1 + |Score|).
•	If the activity was shown within the last 2 hours, reduce that multiplier magnitude by 50%, but never below 1.
o	In other words, the effective multiplier magnitude becomes max(1, (1 + |Score|) / 2).
•	Rebuild the score-based weight using this reduced magnitude:
o	If Score is positive or zero, the weight becomes that reduced magnitude.
o	If Score is negative, the weight becomes the inverse of that reduced magnitude.
This ensures an example like: Score +3 normally gives 4×, but if shown within 2 hours it becomes 2×. It also symmetrically softens negative penalties (e.g., -3 becomes 1/2 instead of 1/4).
5.	Apply the anti-repetition penalty for the last 4 activities:
•	If the activity’s ID is present in RecentList (meaning it appeared in the last 4 selections), multiply its current weight by 0.1 (10× less likely).
6.	Ensure weights are never zero:
•	If needed, enforce a very small minimum weight (for example, 0.0001) so every activity remains theoretically selectable.
7.	Perform the selection using weighted random choice:
•	Compute the final weight for every activity using steps 3–6.
•	Sum all final weights.
•	Pick a random number from 0 up to that sum.
•	Walk through the activities accumulating weights; the first activity whose cumulative weight exceeds the random number is selected.
This guarantees each activity’s chance is proportional to its final weight.
8.	After an activity is selected:
•	Set its LastShownAt to the current time.
•	Insert its ID at the front of RecentList.
•	If RecentList exceeds 4 items, remove the oldest one so the list stays length 4.

Activity selection uses weighted randomness from enabled categories only. Unselected categories do not appear in the results.

This system must:
- Work offline
- Persist across app restarts
- Never reduce probability to zero
-Include both activities from our database and the user created ones.

NAMING LOGIC

HappyPause Image Naming Convention Specification
Each image filename is only a 13-digit number:
XXXXXXXXXXXXX.png
This number is the unique Activity ID.
________________________________________
13-Digit ID Structure
[BATCH][CUSTOM_FLAG][CATEGORY_CODE][ACTIVITY_SEQUENCE]
Breakdown
• BATCH (1 digit)
Represents the content batch version.
Current batch = 1.
Future batches may be 2, 3, etc.
• CUSTOM_FLAG (1 digit)
0 = default activity
1 = custom user-created activity
• CATEGORY_CODE (3 digits)
001 = FITNESS
002 = LEISURE
003 = SOCIAL
004 = MIND
005 = SPIRITUAL
006 = RELAXATION
Future Categories will be added later
• ACTIVITY_SEQUENCE (8 digits)
Sequential number of the activity within its category.
________________________________________
Example
Filename:
1000100000001.png
Breakdown:
1 0 001 00000001
│ │ │   └────── Activity #1
│ │ └────────── FITNESS category
│ └──────────── Default activity
└────────────── Batch 1
Meaning:
•	Batch 1
•	Default activity
•	Category FITNESS
•	First activity in that category
________________________________________
System Instructions
When loading an activity:
1.	Extract the 13-digit number from the filename (remove .png).
2.	Use it as the unique Activity ID.
3.	Match it to the database record with the same ID.
4.	Digit 1 → Batch number
5.	Digit 2 → Custom or default
6.	Digits 3–5 → Category
7.	Digits 6–13 → Activity sequence
The filename ID must always match the database ID exactly.
________________________________________
7. Why This Structure Matters
This system:
•	Avoids duplicate images
•	Creates unique code for each activity
•	Allows infinite scaling (up to 99,999,999 activities per category)
•	Separates system content from user-generated content
•	Makes the filename itself a lightweight database key
________________________________________
8. Important Rule
The filename ID MUST always match the database ID.
The app should treat the 13-digit number as the single source of truth.
