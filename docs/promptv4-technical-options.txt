1. MAIN FRONTEND STACK (MOBILE + WEB)
================================================================================

CHOSEN OPTIONS:
- Mobile: React Native (iOS + Android)
- Web: React (same language/family as mobile)
- Approach: Shared logic, separate codebases (or React Native Web for same codebase)

OTHER OPTIONS (not chosen):

MOBILE:
- Flutter (Dart) – Single codebase for iOS + Android, different language
- Kotlin Multiplatform – Native performance, shares logic across platforms
- Swift/SwiftUI (iOS only) + Kotlin (Android only) – Native, separate codebases

WEB:
- Vue.js – Gentler learning curve, good ecosystem
- Svelte/SvelteKit – Less boilerplate, fast
- Angular – Enterprise-focused, more structure
- Solid.js – React-like, fine-grained reactivity

UNIFIED (same codebase for mobile + web):
- React Native Web – React Native runs in browser; one codebase for all
- Flutter Web – Flutter compiles to web; one codebase for mobile + web
- Tauri + web – Desktop + web; mobile needs separate solution

NOTES FOR NON-DEVS:
- "Main stack" = the technology used to build the user interface (what users see
  and tap/click). Different stacks have different tradeoffs.
- React / React Native = JavaScript/TypeScript. React for web, React Native for
  mobile. Same language, similar patterns. Large ecosystem, many developers.
- Flutter = Google's framework. Uses Dart. One codebase for iOS, Android, and web.
  Fast, polished UI. Different language from typical web.
- "Unified" = one codebase for mobile and web. Fewer codebases to maintain, but
  web experience may feel less "web-native."
- "Separate codebases" = different projects for mobile and web. More work, but
  each platform can be optimized. Shared backend (API) handles data.

DECISION FACTORS:
- Need same codebase? → React Native Web or Flutter
- Prefer JavaScript/TypeScript? → React / React Native
- Want maximum performance/native feel? → Flutter or native (Swift/Kotlin)
- Web-first, mobile later? → React, Vue, or Svelte for web; add React Native
  or Flutter when you build mobile

STACK PAIRINGS (mobile ↔ web):
- React Native + React → Same language, shared components possible via React Native Web
- Flutter + Flutter Web → One codebase, Dart for both
- React Native + Next.js/Remix → React ecosystem, separate projects
- Vue + Capacitor/Ionic → Web framework + native wrapper for mobile

2. BACKEND / API STACK
================================================================================

CHOSEN OPTIONS:
- Server stack: Node.js + Express
- Database: PostgreSQL (for both development and production)
- API: REST API with CRUD operations
- Endpoint list: Use REST conventions (infer from spec)
- API base URL:
  - Dev: localhost
  - Prod: No prod URL yet, or placeholder (e.g. happypausetime.mobileappslabs.ca)
- Env vars: DATABASE_URL, JWT_SECRET, PORT (and Google/Apple client IDs when configured)

OTHER OPTIONS (not chosen):
- Server: Python (FastAPI/Django), Go (Gin/Fiber), .NET (ASP.NET Core)
- Database: MySQL, SQLite (simpler but less scalable)

NOTES FOR NON-DEVS:
- "Backend" = the server that stores your data and handles login, activities, etc.
- Node.js = JavaScript on the server; common choice for React Native projects
  because it uses the same language.
- PostgreSQL = the database that stores users, activities, categories, events.
- REST API = a standard way for the app to talk to the server (get data, save data).
- REST conventions = standard endpoint patterns (e.g. GET /activities, POST /auth/login).

4. REACT NATIVE STACK (mobile-specific)
================================================================================

CHOSEN OPTIONS:
- Framework: Expo
- Navigation: React Navigation (tabs, stacks, modals)
- State management: React Context + useState/useReducer
- Local storage: AsyncStorage
- RN version: Latest stable build

OTHER OPTIONS (not chosen):
- Framework: Bare React Native (no Expo)
- Navigation: Expo Router (file-based), React Native Navigation (Wix)
- State management: Zustand, Redux Toolkit, Jotai, MobX
- Local storage: MMKV (faster), SQLite, WatermelonDB
- RN version: 0.74 LTS (older, more stable)

NOTES FOR NON-DEVS:
- Expo = a toolchain that simplifies React Native development and builds.
- "Navigation" = how users move between screens (Timer, Stats, History, Profile).
- "State management" = where app data lives in memory (e.g. timer state, user info).
  Context + useState/useReducer = built-in React tools, no extra library.
- AsyncStorage = simple key-value storage for settings, tokens, etc.

5. OAUTH / SOCIAL LOGIN
================================================================================

CHOSEN OPTIONS:
- Google: @react-native-google-signin/google-signin
- Apple: @invertase/react-native-apple-authentication (or expo-apple-authentication)
- Both Google and Apple sign-in enabled
- Token flow: App sends ID token to backend → backend verifies → returns our JWT
- Token storage: SecureStore (Expo)

OTHER OPTIONS (not chosen):
- Google only or Apple only
- Firebase Auth (simpler but ties you to Firebase)
- Token storage: AsyncStorage (simpler but not encrypted)

NOTES FOR NON-DEVS:
- "OAuth" = "Sign in with Google" or "Sign in with Apple" instead of
  email/password.
- Both Google and Apple sign-in are implemented.
- You need: Google Cloud Console (client IDs), Apple Developer account (for Apple).
- "JWT" = a secure token that proves the user is logged in. The app stores it
  and sends it with every request.
- SecureStore = encrypted storage for tokens (Expo built-in).

7. DATA SYNC
================================================================================

CHOSEN OPTIONS:
- Sync logic: Server-first throughout. PostgreSQL is the source of truth.
  User-created activities and events: queue when offline, push to server
  when online.
- Conflict resolution: Server-wins for server activities; last-write-wins for
  user-created.
- Sync timing: On app start + when returning to foreground.

OTHER OPTIONS (not chosen):
- Sync logic: Server-only (no offline), always-online required
- Conflict resolution: Client-wins, merge, prompt user
- Sync timing: Periodic (e.g. every 15 min), real-time (WebSocket)

NOTES FOR NON-DEVS:
- "Sync" = keeping data on the device and server in agreement.
- "Server-first" = PostgreSQL is the source of truth; app fetches and caches.
- Offline support = app queues writes locally when offline, pushes to server
  when back online.
- "Conflict" = when the same activity was edited both offline and on the server.
  - Server-wins = server data overwrites local.
  - Last-write-wins = newest change wins.
- "Append-only" = events (focus_started, thumb_up, etc.) are never edited; we
  just add new ones, so no conflicts.

8. SECURITY
================================================================================

CHOSEN OPTIONS:
- Password hashing: bcrypt (cost factor 10–12)
- JWT expiry: 7 days
- Refresh tokens: No refresh (user re-logs when JWT expires)
- Token storage: SecureStore (Expo)

OTHER OPTIONS (not chosen):
- Password hashing: argon2, scrypt, PBKDF2
- JWT expiry: 1 hour, 24 hours, 30 days
- Refresh tokens: Yes (for fewer logins)
- Token storage: AsyncStorage (not encrypted), react-native-keychain

NOTES FOR NON-DEVS:
- "Password hashing" = passwords are never stored as plain text. We store a
  one-way hash so even we can't recover the original password.
- bcrypt = industry standard for password hashing.
- "JWT expiry" = how long the login token is valid before the user must log in
  again. 7 days = good balance for mobile.
- "Refresh token" = a long-lived token that lets the app get a new JWT without
  asking the user to log in again.
- Secure storage = tokens are stored encrypted on the device.
